## 7. Risk Surface & Files to Change
If extending the schema to include `kts_fehler` (boolean) and `kts_fehler_beschreibung` (string | null), the following files must change:
- **Database:** A new Supabase migration must be created to `ALTER TABLE trips ADD COLUMN kts_fehler boolean, ADD COLUMN kts_fehler_beschreibung text`.
- **Types:** `src/types/database.types.ts` (regenerated).
- **Zod Schemas:** `src/features/trips/components/create-trip/schema.ts` to include the new fields.
- **Create Trip Form:** `src/features/trips/components/create-trip/create-trip-form.tsx` and `sections/payer-section.tsx` to handle the new inputs and state.
- **Trip Detail Sheet:** `src/features/trips/trip-detail-sheet/trip-detail-sheet.tsx` to add editable fields for the error state and description.
- **Data Tables:** `src/features/trips/components/trips-tables/columns.tsx` to display an error indicator/badge when `kts_fehler` is true.
- **API/Sync:** Any patch building logic (e.g., `buildPartnerSyncPatchFromDrafts`) must include the new fields.

## 8. Senior Recommendation
**Recommendation: Single Migration (Columns on `trips` table)**
Given that `kts_document_applies` and `kts_source` are already scalar fields on the `trips` table, adding `kts_fehler` (boolean) and `kts_fehler_beschreibung` (text) directly to the `trips` table via a single migration is the most pragmatic approach.

- **Why not a related table?** KTS status is tightly coupled to the trip itself. A 1:1 related table would introduce unnecessary join overhead for a feature that is fundamentally a state flag on the trip.
- **Component Reuse:** The existing `PayerSection` and `TripDetailSheet` payer sections should be extended to conditionally show a `Textarea` or `Input` for the description when `kts_fehler` is toggled true.
- **PDF Impact:** The PDF generation code must be updated. If `kts_fehler` is true, the print templates should prominently warn the driver (e.g., "KTS FEHLT!" or "KTS KORRIGIEREN") so they know to actively resolve the issue during the trip.
